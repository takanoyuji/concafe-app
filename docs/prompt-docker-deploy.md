# プロンプト: Docker でローカル起動 → サーバーにデプロイ

別プロジェクトで「Docker でローカル立ち上げてからサーバにアップする」運用を依頼するときに、以下をコピーして使ってください。

---

## プロンプト（コピー用）

```
このプロジェクトを、次の流れで運用できるようにしてほしい。

1. 手元の PC（WSL またはローカル）で Docker を使ってビルド・起動できるようにする
   - ローカルで動作確認してからサーバに反映したい（サーバ上でビルドすると時間がかかるため）

2. 手元でビルドした Docker イメージをサーバに渡す方法を用意する
   - 方法A: Docker Hub（または GHCR 等）に push して、サーバでは pull のみ
   - 方法B: docker save で tar に出力し、scp でサーバに送って docker load
   - どちらか、または両方の手順を README や DEPLOY.md に書いてほしい

3. サーバ側では「ビルドせず、イメージの pull（または load）と up だけ」で起動できるようにする
   - docker-compose.pull.yml のような「イメージ指定のみ」の compose ファイルがあるとよい

4. データ永続化
   - DB やアップロードファイルなど、消えて困るデータは Docker のボリュームに保存するようにする
   - コンテナを再作成（--force-recreate）してもデータが消えないようにしてほしい
   - デプロイ前のバックアップ手順も README に書いてほしい

5. 環境変数
   - ビルド時に埋め込む必要があるもの（例: NEXT_PUBLIC_*）は .env から読んでビルドに渡すようにする
   - 手順に「事前に .env に ○○ を入れておく」と明記してほしい

以上を、このリポジトリの構成に合わせて実装・ドキュメント化してほしい。
```

---

## プロジェクト固有の情報を足すときの例

- **レジストリ**: 「Docker Hub のユーザー名は ○○ を使う」
- **サーバ**: 「サーバは Ubuntu、SSH は `ssh user@example.com`、アプリのパスは `/opt/apps/○○`」
- **イメージ名**: 「push するイメージ名は `○○/プロジェクト名:latest` で」

必要に応じてプロンプトの末尾に追記してください。
